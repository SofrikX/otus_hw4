# Firebase Security Rules — PetConnect HW5

## Назначение

Документ описывает Firestore Security Rules для PetConnect и объясняет, как они заменяют Supabase Row Level Security из оригинального текста ДЗ.

PetConnect использует Firebase Auth и Cloud Firestore. Идентификатор пользователя берется из `request.auth.uid`, а доступ к документам ограничивается правилами в `firestore.rules`.

## Firebase Security Rules вместо Supabase RLS

В Supabase RLS доступ к строкам PostgreSQL ограничивается SQL-политиками. В Firebase аналогичную роль выполняют Security Rules:

- rules применяются на уровне коллекций и документов Firestore;
- `request.auth.uid` заменяет SQL-проверку текущего пользователя;
- выражения `isOwner(ownerId)` и `isAuthor(authorId)` заменяют owner/author policies;
- правила для `chats/{chatId}` и `messages` ограничивают доступ участниками чата;
- правила для вложенных коллекций `comments` и `likes` проверяют автора и родительский документ.

Security Rules не заменяют бизнес-логику полностью. Операции со счетчиками, транзакциями и сложной валидацией лучше выполнять через Cloud Functions.

## Helper Functions

В `firestore.rules` используются helper-функции:

| Helper | Назначение |
|---|---|
| `signedIn()` | Проверяет, что пользователь авторизован |
| `isOwner(ownerId)` | Проверяет, что `ownerId` совпадает с `request.auth.uid` |
| `isAuthor(authorId)` | Проверяет, что `authorId` совпадает с `request.auth.uid` |
| `isChatParticipant()` | Проверяет, что текущий пользователь входит в `participantIds` чата |

Дополнительные helpers проверяют неизменность защищенных полей, корректность комментариев, join-сценарий прогулки и неизменность участников чата.

## Разрешенные операции

### users

- Авторизованный пользователь может читать базовые профили пользователей.
- Пользователь может создать только свой документ `users/{uid}`.
- Пользователь может обновить только свой документ.
- Пользователь может удалить только свой документ.

### pets

- Все авторизованные пользователи могут читать питомцев.
- Авторизованный пользователь может создать питомца, если `ownerId == request.auth.uid`.
- Обновлять и удалять питомца может только владелец.
- При обновлении нельзя поменять `ownerId`.

### posts

- Все авторизованные пользователи могут читать неудаленные посты.
- Создавать пост может авторизованный пользователь.
- `authorId` при создании должен совпадать с `request.auth.uid`.
- Обновлять и удалять пост может только автор.
- `likesCount` и `commentsCount` нельзя менять напрямую клиентским update.

### posts/{postId}/comments

- Все авторизованные пользователи могут читать комментарии.
- Создавать комментарий может авторизованный пользователь.
- `authorId` должен совпадать с `request.auth.uid`.
- Текст комментария должен быть непустым и не длиннее 500 символов.
- Удалять комментарий может автор комментария или автор родительского поста.

### posts/{postId}/likes

- Все авторизованные пользователи могут читать лайки.
- Пользователь может создать или удалить только свой лайк `likes/{uid}`.
- Обновление лайка запрещено.

### walks

- Все авторизованные пользователи могут читать прогулки.
- Создавать прогулку может авторизованный пользователь, если `creatorId == request.auth.uid`.
- Организатор должен быть участником создаваемой прогулки.
- Обновлять прогулку может создатель.
- Участник может выполнить join-сценарий: добавить свой UID в `participantIds` и увеличить `participantsCount` на 1.
- Удалять прогулку может только создатель.

### chats и messages

- Читать чат могут только участники из `participantIds`.
- Создать чат может авторизованный пользователь, если он входит в `participantIds`.
- Обновлять metadata чата могут только участники, не меняя список участников.
- Читать и создавать сообщения могут только участники чата.
- `senderId` сообщения должен совпадать с `request.auth.uid`.
- Обновление и удаление сообщений клиентом запрещено.

## Запрещенные операции

- Любой доступ неавторизованного пользователя к данным приложения.
- Создание или обновление чужого `users/{uid}`.
- Создание питомца с чужим `ownerId`.
- Смена `ownerId`, `authorId`, `creatorId` или `participantIds` вне разрешенных сценариев.
- Прямое изменение `likesCount`, `commentsCount` клиентом.
- Создание комментария от имени другого пользователя.
- Удаление комментария пользователем, который не является автором комментария или автором поста.
- Чтение и запись чатов пользователем, который не входит в `participantIds`.
- Обновление или удаление сообщений клиентом.

## Security Assumptions

- Firebase Auth включен и является источником пользовательской идентичности.
- Клиент не считается доверенной средой.
- Защищенные счетчики и сложные операции должны постепенно переноситься в Cloud Functions.
- Firestore documents должны соответствовать схеме из `docs/firestore_schema.md`.
- Rules нужно дополнить emulator-тестами перед production deploy.

## Проверка

После установки Firebase CLI и добавления backend tests правила нужно проверять через Emulator Suite:

```bash
firebase emulators:exec "npm test --prefix functions"
```

Если tests еще не созданы, минимум нужно вручную проверить сценарии чтения/записи через Emulator UI.
