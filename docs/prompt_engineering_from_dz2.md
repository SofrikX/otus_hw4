# Prompt Engineering from DZ 2 — адаптация под Codex

Этот документ заменяет набор разрозненных файлов из ДЗ 2. В текущем ДЗ используются не все старые материалы, а только техники, которые реально помогают разработке frontend MVP через OpenAI Codex.

## 1. Используемые техники

### Role Prompting

Каждая задача начинается с роли:

```markdown
# Role
Ты Senior Flutter Developer и OpenAI Codex AI coding agent.
```

Это задает уровень ответа и ожидаемую область экспертизы.

### RTCF

Основной формат промптов:

- Role — роль агента;
- Task — конкретная задача;
- Context — файлы, стек, ограничения;
- Format — формат результата.

### Chain of Thought как декомпозиция задачи

В отчете не нужно раскрывать внутренние рассуждения модели. Но полезно просить Codex разбивать задачу на проверяемые шаги:

1. изучить документы;
2. предложить план;
3. изменить файлы;
4. запустить проверки;
5. обновить отчет.

### Iterative Refinement

Каждый крупный экран улучшается итерациями:

1. базовая структура;
2. состояния loading/error/empty/success;
3. адаптивность;
4. тесты;
5. рефакторинг.

### AI-assisted Debugging

Ошибки из терминала передаются Codex с просьбой:

- определить причину;
- предложить минимальное исправление;
- внести изменение;
- повторить проверку;
- зафиксировать проблему в отчете.

### Multimodal Debugging

Скриншоты интерфейса передаются Codex для анализа:

- проблем адаптивности;
- переполнений;
- нечитаемых текстов;
- слабой визуальной иерархии;
- несоответствия UI-концепции.

## 2. Шаблон промпта для разработки компонента

```markdown
# Role
Ты Senior Flutter Developer и OpenAI Codex AI coding agent.

# Task
Реализуй компонент <component_name> для PetConnect.

# Context
Прочитай:
- AGENTS.md
- docs/documents_index.md
- docs/current_homework_scope.md
- docs/user_stories.md
- docs/ui_concepts/ui_description.md

Стек: Flutter, Dart, Riverpod, go_router, Material 3.
MVP работает на mock-данных.

# Requirements
1. Реализуй базовую функциональность.
2. Добавь состояния loading/error/empty/success, если применимо.
3. Используй существующие shared widgets.
4. Не подключай Firebase.
5. Добавь или обнови тесты.
6. Обнови prompts.md и development_report.md.

# Format
1. Какие файлы просмотрены.
2. Какие файлы изменены.
3. Что реализовано.
4. Какие команды проверки выполнить.
```

## 3. Шаблон промпта для тестов

```markdown
# Role
Ты Flutter Test Engineer.

# Task
Проверь и улучши тесты для <feature_name>.

# Context
В ДЗ требуется минимум 3 автоматических теста. Проект использует flutter_test и mocktail.

# Requirements
1. Проверь позитивные сценарии.
2. Проверь error или empty state.
3. Не удаляй тесты ради прохождения.
4. Запусти flutter test или укажи команду для запуска.
5. Обнови development_report.md и prompts.md.

# Format
Верни список тестов, измененные файлы и результат проверки.
```

## 4. Шаблон промпта для отладки

```markdown
# Role
Ты Flutter Debugger.

# Task
Проанализируй ошибку и предложи исправление.

# Context
Команда: <command>
Ошибка:
<paste_console_error>

# Requirements
1. Найди вероятную причину.
2. Предложи минимальное исправление.
3. Не переписывай несвязанный код.
4. После исправления повтори проверку.
5. Добавь кейс в development_report.md.

# Format
Причина → исправление → файлы → команда проверки.
```

## 5. Шаблон промпта для анализа скриншота

```markdown
# Role
Ты Flutter UI/UX Debugger.

# Task
Проанализируй скриншот PetConnect и найди проблемы адаптивной верстки.

# Context
UI-концепция: яркая социальная сеть.
Приложение должно корректно выглядеть на desktop и mobile.

# Requirements
1. Найди визуальные проблемы.
2. Объясни причину.
3. Предложи точечные изменения.
4. После подтверждения внеси изменения.
5. Обнови development_report.md.

# Format
Проблема → причина → файл → исправление → проверка.
```
